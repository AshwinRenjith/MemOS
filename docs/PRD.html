📝 Canonical Source of Truth — Event-Sourced Commit Log

sequenceDiagram participant SDK as Agent SDK participant GW as Memory Gateway participant PG as PostgreSQL participant CDC as Debezium participant Kafka as Kafka participant Qdrant as Qdrant Writer participant Neo4j as Neo4j Writer SDK->>GW: POST /memory (content, branch) GW->>PG: BEGIN TRANSACTION PG->>PG: INSERT commits table PG->>PG: INSERT outbox table PG->>GW: COMMIT TRANSACTION GW->>SDK: 202 Accepted (commit_hash) CDC->>Kafka: Publish commit event Kafka->>Qdrant: Consume & write embeddings Kafka->>Neo4j: Consume & update graph Note over Qdrant,Neo4j: Idempotency: commit_hash as key

🔑 Key Design Principles:

  • ✓ Atomic Transaction: COMMIT record + OUTBOX event written in same PostgreSQL transaction
  • ✓ CDC via Debezium: Change Data Capture picks up outbox events and publishes to Kafka
  • ✓ Idempotency: Each consumer tracks processed commit_hash. Duplicates are no-ops.
  • ✓ Recovery: If any projection store fails, replay Kafka topic from offset 0 for bit-identical state

💾 Four-Tier Memory Storage Architecture

⚡ Tier 1: Redis (Working Memory)

Low-latency real-time access for active session context

Response: <5ms
Use Case: Current conversation context, temporary state
Retention: Session-scoped, cleared on disconnect

🎯 Tier 2: Qdrant (Episodic Memory)

Vector embeddings for semantic similarity search

Vector Size: 1536 (OpenAI) / 1024 (BAAI)
Index: HNSW with cosine similarity
Use Case: Semantic retrieval, near-duplicate detection

🕸️ Tier 3: Neo4j (Semantic Knowledge Graph)

Graph relationships between entities, facts, and beliefs

Node Types: ENTITY, FACT, RULE, BELIEF
Edge Types: RELATED, SUPPORTS, CONTRADICTS
Use Case: Relationship reasoning, entity linking

📊 Tier 4: PostgreSQL (Relational Memory)

Structured facts with ACID transaction guarantees

Schema: Full DDL in §5.1
Isolation: Row-Level Security (RLS) per org
Use Case: Structured queries, audit trail, compliance

🔢 Step-by-Step Write Process

1

Agent SDK → Memory Gateway

Agent calls POST /v1/repos/{repo_id}/memory with content, source_type, priority, branch, evidence, data_class

{
  "content": "User prefers Python for data tasks",
  "source_type": "USER_STATED",
  "priority": "normal",
  "branch": "main",
  "data_class": "GENERAL"
}
2

Intent Classification + Poisoning Defense

Intent classifier evaluates for injection attacks. If suspicious → route to SANDBOX branch

≥95% attack detection ≤2% false-block rate
3

Atomic Transaction: Commit + Outbox

Within single PostgreSQL transaction:

  • • Write COMMIT record to commits table
  • • Write OUTBOX event to outbox table
BEGIN TRANSACTION;
  INSERT INTO commits (...) VALUES (...);
  INSERT INTO outbox (aggregate_id, event_type, payload) VALUES (...);
COMMIT TRANSACTION;
4

CDC via Debezium → Kafka

Debezium captures outbox changes and publishes to Kafka topic: memory.commits.{org_id}

Topic: memory.commits.org_abc123
Key: commit_hash
Value: {event_type, payload, timestamp}
5

Projection Writers (Idempotent)

Multiple consumers process Kafka events in parallel:

  • • Qdrant Writer: Generate embeddings, store in episodes_{org_id}
  • • Neo4j Writer: Create/update graph nodes and edges
  • • Redis Invalidator: Bust relevant cache entries
Idempotency key = commit_hash Duplicates = no-ops

🔬 Unified Semantic Diff Engine

Contradiction Detection Pipeline (P0 Fix Applied)

1 Embed: Generate embedding for new memory candidate
2 ANN Retrieval: Get top-20 candidates with cosine ≥ 0.50
3 Classification:
• sim ≥ 0.95 → DUPLICATE (reinforce existing)
• sim 0.85-0.95 + NLI ENTAILMENT → REFINEMENT
• sim 0.85-0.95 + NLI CONTRADICTION → CONTRADICTION (create ConflictRecord)
• sim 0.60-0.85 → Run NLI for non-obvious contradictions
• sim 0.50-0.60 → WEAKLY_RELATED (add graph edge)
• No candidates → NOVEL
4 Conflict Escalation: If CONTRADICTION → write to SANDBOX branch, create ConflictRecord

⚠️ P0 Fix Applied: Cosine-sign-based contradiction (cosine < 0.00) is REMOVED. All contradiction detection goes through the unified ANN → NLI classifier path. This is deterministic and auditable.

🎯 Scoring Fusion Engine

📍 Relevance Score

Cosine similarity between query embedding and memory embedding

Range: 0.0 - 1.0
Source: Qdrant ANN search
Weight: 40% of final score

⏰ Recency Score

Time decay function based on commit timestamp

Range: 0.0 - 1.0
Formula: exp(-λ × days_since_commit)
Weight: 30% of final score

⭐ Importance Score

ML classifier prediction + access count boost

Range: 0.0 - 1.0
Model: F1 ≥ 0.78 on high-importance class
Weight: 30% of final score

Final Score Calculation:

final_score = (relevance × 0.40) + (recency × 0.30) + (importance × 0.30)

// Memories ranked by final_score, filtered by min_confidence and min_importance
// Top results assembled into XML context block for LLM injection

⏱️ Time-Travel Retrieval (Audit Feature)

How It Works:

1.
Query with as_of_commit parameter: 
GET /v1/repos/{repo_id}/retrieve?as_of_commit=abc123...
2.
System replays commits 0 → as_of_commit:
  • • Load all commits up to and including target hash
  • • Apply each commit's diff_object in order
  • • Build memory state at that point in time
3.
Acceptance Test (INV-03):
Given commit hash T, retrieval query at as_of=T returns identical result set in 3 independent runs

🎯 Use Case: When your agent makes a wrong decision, you can audit exactly what it believed at that moment by querying memory state at the commit hash corresponding to that decision point.

🔄 Lifecycle State Machines

🌿 Branch Lifecycle State Machine (P0 Fix Applied)

⚠️ P0 Fix — Branch-Delete Audit Contradiction: v1.0 contradiction: 'no garbage data persists' vs 'compliance can inspect abandoned branches'. Resolution: soft-delete separates operational state from data retention. Commit data is ALWAYS retained per retention policy regardless of branch operational status.

stateDiagram-v2 [*] --> ACTIVE ACTIVE --> SOFT_DELETED: agent/user deletes ACTIVE --> MERGED: PR approved + merged ACTIVE --> LEGAL_HOLD: legal hold request SOFT_DELETED --> HARD_DELETED: retention TTL expires AND no legal hold SOFT_DELETED --> LEGAL_HOLD: legal hold request LEGAL_HOLD --> SOFT_DELETED: hold expires + release MERGED --> [*]: terminal state HARD_DELETED --> [*]: terminal state LEGAL_HOLD --> [*]: terminal state (immovable) note right of ACTIVE Normal operational state Visible in listings Can be mutated end note note right of SOFT_DELETED Not visible in normal listing Commits accessible via audit API Retention period active end note note right of LEGAL_HOLD Immutable Cannot be deleted Until hold released end note

🎯 Use Case: Ensures compliance with data retention policies while maintaining operational flexibility. Branches can be soft-deleted for cleanup but remain auditable until retention expires.
