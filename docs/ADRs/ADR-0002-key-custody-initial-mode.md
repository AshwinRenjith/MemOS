# ADR-0002: Initial Key Custody Mode

| Field           | Value                                    |
|-----------------|------------------------------------------|
| **Status**      | ACCEPTED                                 |
| **Date**        | 2026-03-11                               |
| **Author**      | AI Builder (Phase 0)                     |
| **Phase**       | Phase 0 (baseline), Phase B (enforcement)|
| **PRD Section** | §7.1 (Three-Tier Key Custody Model)      |
| **Invariants**  | INV-02                                   |

---

## Context

The PRD defines three key custody tiers (§7.1):
1. **HOSTED_KMS** — MemoryOS manages keys via AWS/GCP KMS (Trust Level 3).
2. **CUSTOMER_KMS** — Customer manages their own HSM/Vault (Trust Level 4).
3. **LOCAL_KEY** — Private key generated locally by agent process (Trust Level 2).

Phase A needs cryptographic signing infrastructure to exist so that:
- The commit hash chain can be verified (INV-02).
- The signature verification path is tested end-to-end.
- Phase B can enforce signing without a mid-phase migration.

However, Phase A is a development alpha. We do not yet have KMS integration,
and requiring HOSTED_KMS from Day 1 would block local development.

---

## Decision

### Phase A: LOCAL_KEY as default, signing infrastructure in place

1. **LOCAL_KEY mode is the default** during Phase A development and testing.
2. **Ed25519 keypairs** generated by the SDK/test harness using `PyNaCl`.
3. **Signing and verification code paths are fully implemented** — not stubbed.
   Every commit is signed. Every read verifies signatures. The full hot path exists.
4. **Key Registry table** is created in Phase A schema. Public keys are registered.
5. **`key_custody_mode`** column on `agents` table defaults to `LOCAL_KEY`.

### Phase B: HOSTED_KMS enforcement

1. **HOSTED_KMS** integration added (AWS KMS or GCP Cloud KMS).
2. **Non-test environments** (`staging`, `production`) MUST use `HOSTED_KMS` or `CUSTOMER_KMS`.
3. **LOCAL_KEY agents** are restricted: cannot auto-merge to main (PRD §7.1).
4. The transition is backwards-compatible: old LOCAL_KEY-signed commits remain
   verifiable because old public keys are retained in the Key Registry.

### Phase D: CUSTOMER_KMS

1. Customer-managed key custody for sovereign/enterprise deployments.
2. Requires Key Registry to support importing external public keys.

### Key Lifecycle in Phase A

```
Agent/Test Setup:
  1. Generate Ed25519 keypair (PyNaCl)
  2. Store private key in memory or OS keychain (never plaintext on disk)
  3. Register public key via POST /v1/keys/register
  4. key_custody_mode = LOCAL_KEY
  5. All commits signed with private key
  6. Gateway verifies with registered public key
```

### Key Storage Abstraction

To enable smooth transition from LOCAL_KEY to HOSTED_KMS:

```python
class KeyProvider(Protocol):
    """Abstract key custody interface."""
    async def sign(self, data: bytes) -> bytes: ...
    async def verify(self, data: bytes, signature: bytes, public_key: bytes) -> bool: ...
    async def get_public_key(self, agent_id: UUID) -> bytes: ...

class LocalKeyProvider(KeyProvider): ...   # Phase A — PyNaCl Ed25519
class HostedKMSProvider(KeyProvider): ...  # Phase B — AWS KMS
class CustomerKMSProvider(KeyProvider): ... # Phase D — Vault/HSM
```

The `KeyProvider` protocol is injected via dependency injection. Service code
never directly calls PyNaCl or KMS APIs.

## Alternatives Considered

| Alternative                       | Pros                       | Cons                                      | Why Rejected                               |
|-----------------------------------|----------------------------|-------------------------------------------|--------------------------------------------|
| Stub signing in Phase A           | Simpler initially          | Phase B requires full rewrite             | Violates "signing infrastructure in place" |
| HOSTED_KMS from Day 1             | Highest trust immediately  | Blocks local dev, needs cloud account     | Over-constraining for alpha phase          |
| RSA-2048 instead of Ed25519       | Wider ecosystem support    | Slower, larger signatures                 | Ed25519 is faster, smaller, and modern     |

---

## Consequences

### Positive
- Full signing/verification pipeline tested from Phase A Day 1.
- No Phase B migration required for the code path — only key provider swap.
- LOCAL_KEY restrictions (Phase B) are enforceable because the mode is tracked per-agent.

### Negative / Trade-offs
- LOCAL_KEY has lower trust assurance. Development commits are technically less trustworthy than KMS-signed ones.
- Developers must manage keypairs locally during Phase A.

### Risks
- Key loss during development → agents lose signing ability. Mitigated: Phase A agents are ephemeral/test-only.

---

## Rollback / Migration

- Phase A → Phase B transition requires no data migration: old commits retain LOCAL_KEY signatures, new commits use HOSTED_KMS.
- Key Registry retains all historical public keys (never deleted, per PRD §7.3).
- If HOSTED_KMS proves unreliable: fall back to LOCAL_KEY with a feature flag. Log a degradation warning.

---

## Validation

- **Unit test**: Sign with LOCAL_KEY provider → verify with same provider → pass.
- **Unit test**: Sign with LOCAL_KEY → tamper with payload → verify → fail (tamper detection).
- **Integration test**: Full write → commit → retrieve → verify signature chain.
- **Phase B gate test**: HOSTED_KMS-signed commit verified offline without KMS network call (INV-02).
